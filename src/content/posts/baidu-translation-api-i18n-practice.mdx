> 这篇文章记录一次真实改造：把一个中文博客升级为中/英/日三语站点，并接入百度通用翻译 API，最终做到“可用、可维护、可扩展”。

## 一、改造目标与约束

这次改造的目标并不只是“能翻译”，而是满足下面 4 个工程要求：

1. **路由层可管理**：支持 `/zh`、`/en`、`/ja`，旧路径自动跳转；  
2. **翻译层可替换**：封装 API 调用，未来可切换到其他翻译供应商；  
3. **内容层可稳定**：长文章翻译不能随机失败；  
4. **体验层可接受**：语言切换和页面跳转有反馈、不卡死。

## 二、路由国际化：先把骨架搭好

我采用了基于目录的 locale 路由结构：

- `src/app/[locale]/page.tsx`
- `src/app/[locale]/blog/page.tsx`
- `src/app/[locale]/blog/[slug]/page.tsx`
- `src/app/[locale]/profile/page.tsx`

同时在中间件里做了两件事：

- 旧路径（如 `/blog`）重定向到默认语言（`/zh/blog`）；
- 非 locale 路径统一补默认前缀，减少死链和历史入口问题。

这一步完成后，站点信息架构就稳定了，后续只要关注“每个页面拿什么语言的数据”。

## 三、翻译 API 接入：不是填个 key 就完事

百度通用翻译接口的核心是签名：

`sign = md5(appid + q + salt + secret)`

请求参数包括：

- `q`：待翻译文本
- `from`：源语言（本项目固定 `zh`）
- `to`：目标语言
- `appid`
- `salt`
- `sign`

这次踩到的关键坑有两个：

1. **语言码差异**：站点用 `ja`，但百度目标语言要 `jp`，否则报 `58001 INVALID_TO_PARAM`。  
2. **请求方式问题**：长文本走 GET 容易不稳定，改为 `POST + x-www-form-urlencoded` 后明显稳定。

## 四、正文“部分翻译失败”的根因与修复

最开始出现了典型现象：标题翻译成功，正文仍是中文。  
根因主要是这三类：

- 单次文本太长，接口失败后整段回退；
- 并发请求过高触发频控；
- 页面曾被静态预渲染，失败结果被缓存。

对应修复策略：

### 1）长文分段翻译

把正文按长度切片翻译，再拼接回来，避免“大段一次性失败”。

### 2）保留代码块原文

Markdown 里的 fenced code block（```）不翻译，避免示例代码被破坏。

### 3）失败重试 + 降并发

对可重试错误做指数退避重试，并把列表页翻译并发改为串行，降低被限流概率。

### 4）改为动态渲染路径

让语言页按请求动态取翻译，避免“第一次失败被静态固化”。

## 五、性能优化：让切换不再“卡住”

翻译是远程调用，慢是必然的，关键在于“体感优化”：

- **路由预取**：提前 prefetch 常用页面与语言版本；
- **翻译缓存**：服务端内存缓存 + in-flight 去重，避免同段文本重复请求；
- **全局 Loading**：语言切换和页面跳转都给出可见反馈，降低等待焦虑。

结果是：首次进入某语言页面仍需要等待翻译，但二次访问速度提升明显，用户体验从“像卡住”变成“可预期等待”。

## 六、这次实践的可复用清单

如果你也要做同类改造，建议按这个顺序推进：

1. 先做路由改造与旧链路重定向；
2. 再封装翻译服务（签名、重试、错误日志）；
3. 再处理 Markdown 长文策略（分段、代码块保护）；
4. 最后做性能与交互（缓存、预取、loading）。

这样可以把复杂问题拆成“结构正确 -> 数据可用 -> 体验可接受”三步，不会在一个点上反复打补丁。

## 七、结语

国际化不是“把文案翻成英文”这么简单，它本质上是一次**路由、数据、缓存、交互**的系统改造。  
当翻译 API 真正进入生产路径后，工程化细节（错误处理、缓存策略、降级方案）才是决定稳定性的关键。
